# Operations UI Redesign Design

## 结论

本次改版采用“编辑型运营后台”方案，重做 `/getContentHtml` 与 AI 日报结果页，使产品从“工具脚本页面”升级为“内容筛选与生成工作台”。页面结构以后台的状态清晰度为骨架，以内容产品的阅读体验为核心，不走纯 BI 面板，也不走纯资讯站首页。

本文面向本项目维护者与后续实现者。默认读者已了解当前 Cloudflare Worker 路由与页面由 `src/htmlGenerators.js` 生成的现状。本文不包含具体 CSS 或 JS 实现细节，也不覆盖抓取逻辑、AI 生成逻辑或数据存储改造。

## 目标

- 提升内容选择页的可扫描性、可批量操作性与状态反馈质量。
- 提升 AI 日报结果页的阅读体验与后续操作效率。
- 建立统一的后台视觉语言，覆盖信息层级、按钮优先级、反馈方式与响应式行为。
- 消除当前页面对 `alert`、`confirm`、隐藏操作和长文本堆叠的依赖。

## 非目标

- 不改动现有抓取源、抓取分类和 AI 生成业务逻辑。
- 不引入新的前端框架或组件库。
- 不在本次设计中加入复杂搜索、拖拽排序、多人协作或历史版本管理。
- 不把页面改造成传统左侧导航多模块后台；本次仍聚焦单工作流页面。

## 现状问题

### 内容选择页

- 主要动作同时抢占注意力，`抓取`、`保存 Cookie`、`生成日报` 缺少优先级区分。
- 内容条目以“复选框 + 长文本”方式堆叠，难以快速扫读和确认选择结果。
- 选中态弱，用户难以在长列表中建立“已经挑好了什么”的把握感。
- 交互反馈依赖 `alert` 和 `confirm`，打断流程且不保留上下文。
- 双击分类按钮触发抓取属于隐藏交互，不符合可发现性要求。

### 结果页

- 页面更像原始 Markdown 容器，而不是一个可阅读、可继续操作的产品页面。
- 阅读内容与操作按钮没有形成稳定分区，用户很难理解下一步。
- 缺少结果状态、操作说明和导出语义，按钮仅是功能入口的平铺。
- 内容预览与分析结果区域样式层级弱，页面整体不够“完成态”。

## 设计原则

### 1. 后台骨架，内容优先

- 页面结构采用运营后台的清晰分区。
- 核心阅读区保留足够留白与阅读节奏。
- 不把新闻条目压缩成纯表格行。

### 2. 单主动作原则

- 每个页面只允许一个强主按钮。
- 内容选择页的主按钮是 `生成 AI 日报`。
- 抓取、保存设置、查看原文等均为次级动作。

### 3. 反馈替代打断

- 成功、失败、处理中统一使用页内状态、按钮 loading 和 toast。
- 只对真正不可逆的动作使用确认弹层。
- 页面不再依赖 `alert` 和 `confirm` 作为主要反馈方式。

### 4. 选中结果始终可见

- 用户在长列表中做选择时，必须持续看到已选状态。
- 已选摘要不能依赖回滚页面重新确认。
- 桌面端使用固定摘要侧栏，移动端降级为底部摘要条或抽屉。

### 5. 风格克制，信息明确

- 视觉风格采用中性色纸感背景、清晰分割线、低强度阴影。
- 结构借鉴后台，气质借鉴 editorial 阅读产品。
- 不使用花哨渐变、过度玻璃拟态或情绪化动画。

## 视觉系统

### 风格定位

- 结构方向：`Data-Dense Dashboard`
- 阅读气质：`E-Ink / Paper` 与 `Editorial Grid`
- 推荐结果：后台式信息组织 + 阅读型内容卡片

### 色彩建议

- 主色：蓝色，用于主流程和当前激活状态。
- 强调色：琥珀色，用于已选数量、待处理提示和轻量状态标签。
- 背景：浅纸感灰白，不使用纯白大面积硬底。
- 文本：高对比深灰，不使用浅灰正文。

### 字体建议

- 优先方案：中文使用系统无衬线或 `Noto Sans SC`，保证稳定性。
- 可选增强：标题使用 `Newsreader` 风格的阅读型字体，正文保留无衬线。
- 不采用纯代码风字体组合作为主视觉，以免页面过于技术面板化。

### 图标建议

- 统一使用单一线性图标集。
- 图标仅作为辅助，不承担唯一状态表达。
- 分类、状态、操作按钮均需文字标签。

## 页面方案

### 内容选择页

### 页面目标

帮助用户在多个来源和分类中高效筛选内容，明确当前盘面，完成批量勾选，并平滑进入 AI 生成。

### 信息架构

页面拆为四个区域：

1. 顶部状态栏
2. 分类与过滤区
3. 内容池主区域
4. 已选摘要区

### 1. 顶部状态栏

左侧展示全局上下文：

- 日期
- 数据窗口，例如 `7 天内数据`
- 总条数
- 已选条数
- 最近一次抓取状态

右侧展示动作区：

- 主按钮：`生成 AI 日报`
- 次按钮：`抓取最新数据`
- 次入口：`Cookie 设置`

规则：

- 顶部只保留一个强主按钮。
- `保存 Cookie` 不直接暴露在主操作层，改为设置面板中的动作。
- `抓取最新数据` 显示 loading、完成时间和失败原因。

### 2. 分类与过滤区

分类采用带数量徽标的导航条：

- `新闻 24`
- `论文 6`
- `社交平台 12`

过滤区只保留高频条件：

- `仅看已选`
- 来源筛选
- 时间排序
- 清空筛选

规则：

- 不再使用双击分类按钮触发抓取。
- 分类切换要保留用户已选状态。
- 当前分类必须有明显激活样式，且不只靠颜色。

### 3. 内容池主区域

内容池改为卡片式列表，每张卡片包含：

- 选择控件
- 标题
- 来源与发布时间
- 摘要
- `展开摘要` 或 `收起`
- `查看原文`

可选增强信息：

- 来源标签
- 类型标签
- 简短状态标签，例如 `已选`

规则：

- 卡片整体可点击选中，但仍保留显式复选控件。
- 选中态同时通过边框、背景、勾选图标和摘要区同步体现。
- 展开摘要只影响当前卡片，不触发布局剧烈抖动。
- 卡片按钮最小点击高度不低于 `44px`。

### 4. 已选摘要区

桌面端采用右侧固定栏，持续显示：

- 已选总数
- 按分类的已选数量
- 最近选择的若干条标题
- `清空已选`
- `开始生成`

移动端不保留固定侧栏，改为底部摘要条：

- 默认显示 `已选 N 条`
- 点击展开抽屉查看摘要与操作

规则：

- 摘要区不抢主内容视觉权重。
- 摘要区的 `开始生成` 与顶部主按钮语义一致，避免双重行为差异。
- 当未选择任何内容时，摘要区显示空状态文案和引导动作。

### 空状态与异常状态

- 当前分类无内容：显示空状态卡片与建议操作，例如重新抓取或切换分类。
- 抓取失败：在顶部状态栏下方展示错误横幅，提供 `重试`。
- 生成失败：保留当前选择，不清空用户上下文。

### 结果页

### 页面目标

让结果页成为“阅读、复核、继续操作”的工作台，而不是生成后的纯文本落点。

### 信息架构

页面拆为三个区域：

1. 结果页头部
2. 主阅读区
3. 操作与扩展区

### 1. 结果页头部

左侧展示：

- 页面标题
- 生成日期
- 来源条数
- 当前状态，例如 `已生成` 或 `分析中`

右侧展示操作：

- 主按钮：与当前页面的下一步动作对应，例如 `生成播客脚本`
- 次按钮：`AI 日报分析`
- 次按钮：`预览排版`
- 次入口：返回内容选择

规则：

- 页面顶部仍保持一个强主动作。
- 次按钮采用低一级视觉，不与主按钮竞争。
- 返回入口语义清晰，保证回到上一步仍能恢复选择状态。

### 2. 主阅读区

主阅读区是页面核心，展示生成后的日报正文。

设计要求：

- 使用适合长文本阅读的宽度与行高。
- 标题、摘要、分节、引用、代码块要有明显层级。
- 正文容器不使用过重的“代码块盒子”视觉。

可选增强：

- 目录锚点
- 一键复制全文
- 阅读模式与排版预览切换

### 3. 操作与扩展区

桌面端可采用右侧辅助栏，移动端放在正文下方。

包含内容：

- 播客脚本生成入口
- AI 分析结果区域
- 预览与导出动作
- 调试信息折叠区，例如 Prompt 详情

规则：

- 调试信息默认折叠，避免压制主阅读区。
- AI 分析结果采用独立卡片，异步加载时显示骨架或加载状态。
- 新窗口预览应强化为“排版预览”语义，而不是技术动作名。

## 响应式策略

### 桌面端

- 内容选择页使用 `主内容区 + 右侧摘要栏`
- 结果页使用 `主阅读区 + 右侧操作栏`

### 平板端

- 右侧区域收窄为较轻的辅助列
- 过滤器允许换行，不强制单行撑满

### 移动端

- 摘要区改为底部浮动条或抽屉
- 顶部状态栏改为上下堆叠
- 内容卡片仍保持完整标题与关键元信息
- 所有主要按钮高度不低于 `44px`

## 交互规则

### 必须保留的交互

- 勾选条目
- 查看原文
- 展开摘要
- 按分类切换
- 批量生成

### 必须替换的交互

- `alert` 改为 toast 或页内反馈
- `confirm` 改为必要时的确认弹层
- 双击触发抓取改为显式按钮

### 状态反馈

- 按钮执行中显示 loading 与禁用态
- 成功操作显示短暂成功反馈
- 失败反馈需要说明原因与恢复动作

## 可访问性要求

- 所有交互控件保留可见 focus 样式
- 状态不能只靠颜色表达
- 图标按钮必须带文本或 `aria-label`
- 卡片选中状态需被屏幕阅读器正确感知
- Toast 使用 `aria-live="polite"`
- 展开与收起控件需要可访问状态说明

## 风险与控制

- 风险 1：页面过度后台化，阅读体验下降
  - 控制：主内容继续使用卡片与阅读排版，不用纯表格替代内容
- 风险 2：右侧摘要栏过重，挤压主区
  - 控制：摘要栏只保留关键统计与近期选择，避免长列表复制主内容
- 风险 3：移动端因为双栏思路导致拥挤
  - 控制：移动端改为单栏流 + 底部摘要条
- 风险 4：功能入口增多后主次不清
  - 控制：每页只保留一个强主按钮，其他动作统一降级

## 验收标准

- 内容选择页能在首屏明确展示日期、总条数、已选条数和主动作。
- 用户在长列表中始终能确认当前已选结果。
- 页面不再依赖双击、`alert`、`confirm` 作为核心交互。
- 结果页具备明确的阅读主区和操作次区。
- 桌面端与移动端均有清晰的摘要与操作降级策略。
- 视觉风格统一为“后台结构 + 阅读气质”的混合方案。
