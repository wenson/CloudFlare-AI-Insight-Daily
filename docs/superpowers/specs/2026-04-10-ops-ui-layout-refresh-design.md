# Operations UI Layout Refresh Design

## 结论

本次改版采用“混合型内容工作台”方案，明确以内容筛选与生成流程为主，以运营状态与辅助操作为辅。整体气质延续当前 Worker 后台的稳定性，但视觉上从“工具页面”升级为“编辑型内容工作台”。

相较于 [2026-04-08-ops-ui-redesign-design.md](/Volumes/c/Workspace/CloudFlare-AI-Insight-Daily/docs/superpowers/specs/2026-04-08-ops-ui-redesign-design.md)，本次设计更进一步锁定以下内容：

- 首页采用 `内容编辑台优先，运营状态辅助` 的混合结构。
- 视觉风格采用 `Paper + Control`，即纸感内容产品气质配合后台骨架。
- `Cookie 设置` 与 `Backfill` 统一下沉到默认折叠的 `高级操作` 面板。
- 日报结果页与内容选择页统一成同一套视觉语言与操作优先级。

本文是后续实现与测试的唯一设计依据，聚焦 `src/ui/pageShell.js`、`src/ui/contentSelectionPage.js`、`src/ui/genAiPage.js`、`src/ui/contentArchivePage.js` 这些服务端字符串模板页面，不涉及抓取、鉴权、存储与 AI 生成逻辑变更。

## 目标

- 让 `/getContentHtml` 首屏更清楚地表达“今天有什么内容、已经选了什么、下一步要做什么”。
- 保留抓取、Cookie、Backfill、归档等运维能力，但不让它们抢占主流程注意力。
- 让内容卡片从“长文本堆叠”提升为更易扫描、更易选择、更有完成度的编辑卡片。
- 让日报结果页更像“可阅读、可继续操作”的内容工作台，而不是原始 Markdown 容器。
- 在不引入新框架的前提下，复用现有服务端模板与脚本钩子完成改版。

## 非目标

- 不重写抓取、回填、D1、KV、鉴权或路由处理逻辑。
- 不引入 React、Vue、Tailwind、组件库或构建工具。
- 不加入复杂搜索、多字段过滤、拖拽排序、多人协作或历史版本管理。
- 不改变当前“人工勾选内容后再生成”的产品原则。

## 产品定位

### 核心定位

这是一个“先筛内容，再生成日报/播客”的编辑型工作台，而不是纯运营后台，也不是面向公众消费的资讯首页。

### 设计方向

- 页面结构采用后台式信息组织。
- 页面气质采用内容产品式阅读秩序。
- 用户第一眼应理解内容盘面和主操作，而不是看到一堆管理按钮。

## 页面信息架构

### 1. 内容选择页 `/getContentHtml`

页面结构固定为四层：

1. 顶部状态带
2. 分类与工具带
3. 内容池主区域
4. 已选摘要侧栏

#### 顶部状态带

顶部状态带承担“建立上下文 + 给出主操作”的作用。

左侧信息：

- 页面标题，例如 `2026/4/10 内容工作台`
- 发布日期
- 候选总数
- 已选数量
- 最近一次抓取状态

右侧动作：

- 主按钮：`生成 AI 日报`
- 次按钮：`抓取最新数据`
- 次入口：`高级操作`

规则：

- 首屏只能存在一个强主按钮。
- `抓取最新数据` 保持可见，但视觉优先级低于 `生成 AI 日报`。
- `高级操作` 不直接展开全部辅助功能，只作为受控入口。

#### 分类与工具带

分类与工具带位于状态带下方，采用横向工具条组织。

左侧内容：

- 分类切换按钮，例如 `新闻`、`论文`、`社交平台`
- 各分类数量徽标

右侧内容：

- 每批加载条数切换
- 如有必要可保留极少量轻量筛选，但本轮不新增复杂过滤器

规则：

- 分类切换不丢已选状态。
- 激活态必须同时通过底色、边框或字重体现，不能只靠颜色。
- 批次设置继续沿用现有逻辑，但视觉上纳入统一工具带。

#### 内容池主区域

内容池使用单列卡片流，不使用表格，也不拆成多列瀑布布局。

每张内容卡片包含：

- 显式勾选控件
- 标题
- 来源
- 发布时间
- 摘要
- `查看原文`
- 选中状态标签

规则：

- 支持点击整卡选中，同时保留显式 checkbox。
- 选中态同时通过整卡边框、背景和标签表达。
- 摘要展开与收起只影响当前卡片，不引发大面积重排。
- 无限加载、重试与空状态都保留在列表底部，不提升到全局层级。

#### 已选摘要侧栏

桌面端保留右侧固定摘要栏，是页面第二视觉重心。

摘要栏内容：

- 已选总数
- 按分类的已选数量
- 最近选择的若干标题
- `清空已选`
- `开始生成`

规则：

- 侧栏中的 `开始生成` 与顶部主按钮语义完全一致。
- 侧栏比归档卡更重要，但仍低于主内容池。
- 当未选择内容时，显示空状态说明与引导文案。

#### 辅助区块

辅助区块包括：

- 历史归档
- 高级操作

归档：

- 保留在右侧栏中
- 视觉权重低于已选摘要
- 仍支持快速跳回某一天的内容池

高级操作：

- 默认折叠
- 展开后容纳 `Cookie 设置` 与 `Backfill`
- 不以弹新页面方式呈现

默认策略：

- 首页首屏不直接展示 Cookie 表单与 Backfill 表单
- 它们只在用户明确需要时展开

### 2. 日报结果页

结果页采用与内容选择页一致的视觉语言，结构固定为三块：

1. 结果页头部
2. 主阅读区
3. 操作侧栏

#### 结果页头部

左侧显示：

- 标题
- 日期
- 来源条数
- 当前状态，例如 `已生成` 或 `生成失败`

右侧显示：

- 当前页上下文标签，例如 `结果工作区`

规则：

- 头部仍维持强信息概览，但不放过多按钮。
- 具体操作集中收敛到右侧操作栏。

#### 主阅读区

主阅读区是页面核心，支持两种视图：

- 原稿视图
- 排版预览视图

规则：

- 原稿视图仍允许查看原始 Markdown。
- 排版视图应更像阅读成品，强调标题、分节、引用、图片和段落节奏。
- Markdown 容器不应继续呈现为“代码块盒子”的强技术感样式。

#### 操作侧栏

操作侧栏按优先级排列：

- 与当前页面最相关的主动作，例如 `生成播客脚本`
- `AI 日报分析`
- `预览排版 / 查看原稿`
- 返回内容选择
- 分析面板
- Prompt 面板

规则：

- 保持单主动作原则。
- Prompt 相关内容默认收起，仅在需要时展开。
- 分析结果与提示词记录保留，但弱化其默认视觉侵入性。

## 视觉系统

### 风格定位

最终风格选用 `Paper + Control`：

- `Paper`：提供纸感、内容产品气质、较松的阅读节奏
- `Control`：保留后台结构、状态清晰度和操作效率

不是纯资讯站，也不是纯 BI 面板。

### 色彩

- 主色：蓝色，用于主按钮、激活态、关键操作反馈
- 强调色：琥珀色，用于已选、提醒和轻量状态标签
- 背景：浅纸感灰白与轻蓝渐层，不使用沉重深色或强玻璃效果
- 文本：高对比深灰，保证长文本阅读与状态识别

规则：

- 不新增大面积高饱和颜色区块。
- 成功、失败、处理中状态继续采用语义色，但只作功能强调，不作为整页主基调。

### 排版

- 中文正文继续使用稳定的无衬线系统字体或 `Noto Sans SC`
- 标题通过字号、字重与间距强化层级
- 大数字与状态文案可适度增强，但不引入夸张展示字体

规则：

- 保持移动端正文不少于 16px
- 标题层级清楚，避免不同区域字号随机增长
- 行高优先保证阅读与扫描

### 卡片与阴影

- 主要卡片采用低强度阴影与清晰描边
- 已选态依赖描边、底色和标签组合，不靠重阴影
- 归档卡和高级操作卡采用更轻的层级表现

规则：

- 阴影只作为弱层次辅助手段
- 不做玻璃拟态、不做浮夸霓虹、不做过深投影

## 关键交互

### 内容选择

- 复选框继续保留
- 点击卡片空白区域也可切换选中
- 选中后顶部计数、侧栏摘要、隐藏表单输入同步更新

### 分类切换

- 切换分类不影响已选项缓存
- 当前分类卡片状态与已选状态一起恢复
- 已有分页与懒加载逻辑继续保留

### 无限加载

- 保持使用当前 `IntersectionObserver` 方案
- 列表底部继续承载“继续加载 / 已加载全部 / 重试”状态
- 不新增全局遮罩式 loading

### 高级操作

- `Cookie 设置` 与 `Backfill` 改成默认折叠的高级面板
- 展开时在原位显示，不跳转、不模态化
- 成功与失败继续优先使用页内 toast 和局部反馈

### 结果页交互

- `预览排版` 与 `查看原稿` 继续保留切换机制
- `AI 日报分析` 继续为异步行为，但按钮 loading 与结果面板状态更清楚
- Prompt 展开继续保留显式披露按钮

## 响应式行为

### 桌面端

- 主区与侧栏保持双栏布局
- 侧栏宽度稳定，主内容池优先占据横向空间

### 平板与移动端

- 页面降级为单列
- 侧栏不再长期占据可视空间
- 已选摘要保留为底部入口条
- 高级操作继续默认折叠
- 回到顶部按钮保留

规则：

- 不允许出现横向滚动
- 所有点击目标保持至少 44px 触达面积
- 移动端首屏优先显示状态带、主按钮与内容池起始部分

## 可访问性

- 保留可见 focus 样式
- 激活态不只靠颜色表达
- Toast 区继续使用 `aria-live="polite"`
- 折叠面板继续使用显式按钮和可读标签
- 卡片、按钮、摘要与归档区域维持清晰 heading 层级

## 风险与取舍

### 风险

- 页面模板依赖大量内联字符串与内嵌脚本，布局重排时容易误伤测试选择器
- 将辅助功能折叠后，如果入口设计不够明确，可能影响少数高频运维用户的直达效率
- 结果页视觉升级若处理过重，可能削弱原始 Markdown 可读性

### 取舍

- 本次优先解决层级与完成度，不扩展功能面
- 优先复用现有结构与交互钩子，减少行为回归风险
- 运维效率通过“可达但不抢戏”的方式平衡，而不是让所有操作平铺首屏

## 交付标准

当以下条件同时满足时，视为本次设计实现达标：

- 首页主次关系明显优于当前版本
- 用户能在 3 秒内理解“当前盘面、已选状态、主操作”
- 辅助操作不再干扰主流程，但仍然容易找到
- 日报结果页与内容选择页视觉语言统一
- 移动端仍然可完成选择与生成主流程
